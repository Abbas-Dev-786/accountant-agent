# DEBUG REPORT — N2–N8 production reliability fixes

- **Symptom:** Production Xero sync could read all tenant history and reconcile voided records; Drive collections could fail on multi-parent files; prepared recovery emails exhausted their own rate limit; durable workers could lose leases during slow provider/AI/archive tasks; Plaid pending authorizations matched as posted; and a token refresh reset the browser to the first organization.
- **Root cause:** Xero production client had no close-period input or query filter, reconciliation treated ledger-only exceptions as one Groq request each, Drive chose an arbitrary first parent, email counts included untransmitted actions, the durable SQL store exposed only fixed leases, Plaid's boolean `pending` field was ignored, Xero status was not checked at either boundary, and the mount-only auth listener closed over the first render's organization ID.
- **Fix:** Xero cash reads now require and apply close-period `where` bounds and omit deleted/voided states; the frozen projection also rejects voided/deleted records. Ledger-only exceptions remain visible but are marked `unavailable` for AI explanation, while rejected/unavailable Groq explanations do not block the review package. Drive selects a configured parent. Recovery limits count only succeeded/reconciled sends. The durable worker renews its lease while an executor runs and refuses terminal completion after ownership expires. Plaid `pending: true` wins over a legacy status string. The UI uses refs/session-token gating to retain the selected organization through auth refresh and to avoid duplicate data loads.
- **Regression coverage:** Scoped Xero query, multi-parent Drive response, Plaid pending, voided Xero record, lease renewal/lease loss, Groq unavailability, unmatched-ledger AI exclusion, and recovery-email counting.
- **Evidence:** `backend/.venv/bin/python -m unittest discover -s tests -v` passed — 180 tests. `npm run build` passed in `web`.
- **Status:** DONE
